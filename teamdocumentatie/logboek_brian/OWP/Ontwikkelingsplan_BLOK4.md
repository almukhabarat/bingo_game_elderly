## Blok 1 Project TI: Climate Measurement Box (CMB)

In dit blok heb ik voor het eerst een microcontroller geprogrammeerd met Arduino C++. Dit project heette Climate Measurement Box (CMB) en we moesten dus een temperatuursensor programmeren in combinatie met een webinterface waarop de waardes kunnen worden afgelezen. 

Het was dus de bedoeling dat we aan de slag ging met HTML, CSS en Javascript. Voor HTML en CSS heb ik vooral videos gekeken om de basis te begrijpen. Ik had bijvoorbeeld gekeken hoe een html header tag werkt en hoe ik div tags kon ordenen. Met Javascript leerde ik werken met de webserial API, ik leerde hiermee onderzoek te doen naar het oplossen van een codeprobleem. Hiermee had ik een string die binnenkwam van de microcontroller kunnen opsplitsen in meerdere items die ik dan apart kon manipuleren. Uiteidenlijk konden we de gemeten temperaturen ook naar een database sturen maar hier gingen we in het volgende project mee aan de slag. In dit blok werkten wij ook meteen met de gitlab userstories, deze waren nog wel voor ons al gecreeërd. 

In dit project probeerde ik met een logboek document in markdown mijn leerprogressie bij te houden, mijn leerprogressie had ik ook uiteindelijk op mijn projectwebsite verwerkt. 

Ik was zelf heel tevreden met het eindresultaat, ik denk dat ik een hoog cijfer had gehaald omdat best wel veel dingen over het project in markdown had gedocumenteerd. Een verbeterd punt voor mij misschien was dat ik nogal wat spagetti code had en dat ik misschien iets meer tijd had kunnen besteden aan de database. 

## Blok 2 Project TI: Game Controller

In dit project gingen we voor het eerst samenwerken in duo's. In dit blok moesten wij een game programmeren en een gamecontroller ontwikkelen waarmee de game gespeeld kan worden. Tijdens dit blok werden we voor het eerst geïntroduceerd aan Object Oriënted Programming (OOP), vandaar dat we ook een simpele game moest maken omdat je hierin OOP heel makkelijk kunt toepassen. Zo heb ik bijvoorbeeld encapsulation toegepast bij het schrijven van de code voor de spelerobjecten van onze game. In onze OOP had ik onderzoek gedaan naar alle pijlers van OOP maar het onderwerp inheritance hadden wij nog niet heel uitgebreid toegepast in dit project.

Omdat we in dit blok geïntroduceerd werden aan OOP moesten we dit ook op een bepaalde manier documenteren. We moesten onze code ontwerp gaan documenteren met UML. Ik had dus een UML klassendiagram van onze game in OOP. 

Voor onze game wilden ik ook een scoreboard maken waarin de highscores van de spelers konden worden bijgehouden. Hiervoor moesten wij dus ook een simpele database maken door middel van een simpele database table ontwerp, SQL en een php API. Deze database was nogmaals heel simpel, we hadden geen gebruik gemaakt van meerdere tabellen met complexe relaties met allerlei primary keys, foreign keys etc.

Tijdens dit project vond ik samenwerken wel lastig. Communiceren ging opzich goed, maar ik vond zelf achteraf dat ik misschien iets assertiever moest zijn als ik samen met mijn projectpartner een beslissing moest maken.

Wat ik goed vond gaan in dit project was dat we op een gegeven moment het project goed konden afsluiten ondanks dat hier twijfels over waren. In sprint 2 nam onze productiviteit nogal af omdat we op een gegeven moment vast liepen met wat we nou precies wilden gaan doen.Onze sprint 2 review ging daarom ook niet zo goed. Sprint 3 was dus daarom heel stressvol, maar omdat ik denk dat ik en mijn projectpartner goed met elkaar communiceerde, en dat we uiteindelijk een goed 'reddingsplan' hadden opgezet hadden we toch nog wel een goed resultaat kunnen behalen. 

## Blok 3 Project TI: Optimalisatie Studieomgeving

In dit project gingen we werken met grotere project groepjes. We gingen van een groepje van 2 man, naar een groepje van 4 man. Voor dit project moesten we de studieomgeving op school verbeteren door middel van Internet Of Things (IOT). Tijdens dit project ging ik leren hoe ik een raspberry pi server kon opzetten. Het was de bedoeling dat de database en de website op deze server gehost kon worden. Ik maakte gebruik van het LAMP ontwerp (Linux, Apache webserver, MySQL en PHP) bij het opzetten van de server. Door middel van deze server konden onze microcontrollers (via een API) communiceren met de webserver en de database. 

Tijdens dit project heb ik ook meer onderzoek gedaan aan meerdere onderdelen binnen het project in vergelijking met de vorige 2 blokken. Ik had een klein onderzoekverslag geschreven naar verschillende communicatiemethodes (zoals MQTT) die we konden gebruiken om onze microcontrollers met de webserver te kunnen laten communiceren. In dit onderzoek keek ik naar de voor- en nadelen van een paar communicatieprotocolen, de onderliggende protocolen en hoe deze het best toegepast kunnen worden in microcontrollers. 

Daarnaast had ook een klein onderzoek gedaan naar geluidsensoren die gebruikt konden worden voor ons project. In dit onderzoeksverslag ging ik geluidsensoren en electret microfoons testen in een stilteruimte en de resultaten hiervan met elkaar vergelijken. Ik hield in dit onderzoeksverslag ook de specificaties van elke (geluid)sensor bij. Daarnaast had ik een aansluitschema gemaakt met fritzing en deze gebruikt in mijn onderzoeksverslag.

De feedback die ik vooral kreeg bij m'n onderzoekverslagen was dat het handig is een goede probleemomschrijving te formuleren voordat ik mijn onderzoek volledig uitwerk. Het probleem waar ik een onderzoekverslag voor schreef was in mijn hoofd wel duidelijk, maar ik had dit vaak niet uitgebreid omschreven aan het begin van een onderzoeksverslag en tijdens de beoordeling kreeg ik dat dus terug als feedback. Het is een punt waar ik mij dus extra op ga focussen zodra ik weer een onderzoeksverslag ga schrijven in het volgende blok.

Tenslotte had ik ook wat code voor de microcontroller van mijn geluidsensormodule voor het eerst in OOP geschreven (Arduino C++). Hierbij werkte ik onder andere met header- en cpp bestanden. De classes had ik dan meestal apart in een header bestand verwerkt en de functies hiervan in een CPP bestand. Hierbij had ook meteen alle pijlers van OOP toegepast zoals inheritance en polymorphism. 

Ik vond dit project wel goed gaan, ik moet zeggen dat ik wel al iets assertiever was in vergelijking met het vorige blok. Zeker als je met grotere groepjes gaat werken aan grotere projecten is het belangrijk om aan te geven als je het ergens niet mee eens bent, of als je een bepaalde aanpak of methode wil suggereren. In begin van dit blok had ik er nog wel veel moeite mee namelijk, dit kwam denk ik omdat ik mijn teamgenoten ook nog niet zo goed kende. Daarnaast zou ik ook zeggen dat ik plannen moeilijk vond. Er waren dan wel een aantal userstories uitgewerkt op de gitlab, maar soms hadden we dan nog lange discussie hierover hoe we deze gingen aanpakken. Samenwerken en dan met name taakverdeling was lastig. Soms had je wel iemand in het groepje die soms de neiging had om volledig de leiding over te nemen in het project en daarmee bepaalde taken en verantwoordelijkheden aan te wijzen aan mensen. Het had geen slechte intentie natuurlijk, maar het was soms wel lastig om hierdoor goed met elkaar samen te werken. Ik had dit zelf misschien beter moeten aangeven bij diegene, dat ik dan iets zijn als: "Ik snap wat je wilt doen, maar ik vind het moeilijk om het op die manier te doen, ik zou het liever op deze manier willen doen want zo en zo... en aangezien dat onderdeel wel mijn verantwoordelijkheid is, wat vind je hiervan? etc".

## SMART-leerdoelen voor de aankomende perioden

**Ook wil ik gaan werken aan de volgende 4 SMART-leerdoelen:**

---

### **Werken aan wiskunde vaardigheden**

**Specifiek:** Ik wil mijn wiskunde vaardigheden verbeteren en ook nieuwe wiskunde vaardigheden ontwikkelen zodat ik deze kan toepassen in toekomstige schoolprojecten. 

**Meetbaar:** Ik wil per week 2 uur aan wiskunde door opdrachten te doen in Mathskills op DLO.

**Acceptabel:** Ik ben niet heel goed in algebra maar omdat ik wiskunde B had gedaan op de VAVO denk ik dat ik over voldoende voorkennis beschik om hier aan te kunnen werken.

**Realistisch:** Ik ga eerst met de basis opgaven oefenen in Mathskils om eerst weer m'n voorkennis op te frissen, aangezien ik al meer dan een halfjaar geen algebra heb gedaan.

**Tijdsgebonden:** Aangezien dit het laatste blok is van het schooljaar wil ik zo'n 12 uur aan wiskunde hebben besteed voordat dit blok is afgelopen.

---

### **Verbeteren van Communicatieve Vaardigheden**

**Specifiek:** Ik wil mijn communicatieve vaardigheden verder ontwikkelen omdat ik soms nog het gevoel heb dat mensen soms nog moeilijk vinden om te begrijpen wat ik precies bedoel. Ik heb soms moeite met articuleren wat ik precies bedoel. 

**Meetbaar:** Als ik een gesprek heb met mensen moet ik vaker gaan doorvragen wat ze precies bedoelen en tijdens het gesprek contant aan hen reflecteren wat ik denk dat zij bedoelen zodat zij eventuele misvattingen kunnen corrigeren.

**Acceptabel:** In het algemeen probeer ik hier al sowieso aan te werken in het dagelijks leven en ik denk dat ik over een goede basis beschik omdat ik veel Dr. K op youtube kijk in mijn vrije tijd.

**Realistisch:** Ik wil mensen niet direct gaan overbelastten met een bomvol aan vragen tijdens een project gerelateerd gesprek, maar het is wel belangrijk dat ik af en toe doorvraag tijdens een gesprek als een gesprekspartner een bepaald idee voorlegt bijvoorbeeld.

**Tijdsgebonden:** Aan het eind van dit blok (blok 4) wil ik een feedback formulier naar al mijn teamgenoten hebben opgestuurd waarin staat hoe zij mijn samenwerking hebben ervaren. In dit formulier vraag ik of ze moeite hadden om mij te begrijpen en of zij het gevoel hadden of ik hen goed begreep.
